  <name>Task 1: Add .env auto-load block to backup.sh and restore.sh</name>
  <files>scripts/backup.sh, scripts/restore.sh</files>
  <action>
    Insert the .env loading block immediately after `set -euo pipefail` in both
    scripts. The block must be placed BEFORE the `DATABASE_URL` check in each
    file.

    **scripts/backup.sh** — insert after line 2 (`set -euo pipefail`):

    The file currently starts:
    ```bash
    #!/usr/bin/env bash
    set -euo pipefail

    # Manual backup script for QuantumThreat BTC database
    ```

    Change to:
    ```bash
    #!/usr/bin/env bash
    set -euo pipefail

    if [ -z "${DATABASE_URL:-}" ] && [ -f ".env" ]; then
      set -a
      # shellcheck source=.env
      source .env
      set +a
    fi

    # Manual backup script for QuantumThreat BTC database
    ```

    **scripts/restore.sh** — insert after line 2 (`set -euo pipefail`):

    The file currently starts:
    ```bash
    #!/usr/bin/env bash
    set -euo pipefail

    # Manual restore script for QuantumThreat BTC database
    ```

    Change to:
    ```bash
    #!/usr/bin/env bash
    set -euo pipefail

    if [ -z "${DATABASE_URL:-}" ] && [ -f ".env" ]; then
      set -a
      # shellcheck source=.env
      source .env
      set +a
    fi

    # Manual restore script for QuantumThreat BTC database
    ```

    Rationale for `set -a / set +a`: marks all variables set by `source .env`
    as exported automatically, which is equivalent to what `dotenv/config` does
    for Node.js scripts.

    Rationale for the guard condition: if `DATABASE_URL` is already set in the
    shell environment (e.g. in CI), the .env file is not loaded, preventing
    accidental overrides.

    After editing, run backup from the project root without any manual export:
    ```bash
    ./scripts/backup.sh
    ```

    Confirm a `.dump` file is created in the `backups/` directory. Then verify
    the file is readable:
    ```bash
    DUMP=$(ls -t backups/*.dump | head -1)
    pg_restore --list "$DUMP" | head -5
    ```
